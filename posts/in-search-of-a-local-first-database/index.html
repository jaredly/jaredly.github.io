To start, by my definition of ‚Äúlocal-first‚Äù, a data-storage solution must meet three requirements:

  • My app‚Äôs functionality must be network-optional; accessing & updating my data must work offline, on multiple devices (with syncing that ideally doesn‚Äôt require manual conflict resolution), for extended periods of time, and across browser / device restarts.   • My data must be available. Evaluation Criteria

For the few projects that do make it past the baseline requirements, I‚Äôve got a much longer list of criteria with which to evaluate them. They fall into the categories of Correctness, Cost, and Flexibility.

Correctness

  • How are conflicts handled? Does it require the client (programmer) to write bespoke conflict resolution code?

  • How ‚Äúbullet proof‚Äù is it? How easy is it to get it into a broken state? (e.g. where different clients continue to see inconsistent data dispite syncing)

  • Is there consistency verification built-in, to detect if you‚Äôve somehow gotten out of sync?

  • How well does sync preserve intent? In what cases would a user‚Äôs work be ‚Äúlost‚Äù unexpectedly, or a change that was made ‚Äúlater‚Äù get overridden by a change made before? Can a sync result in data that doesn‚Äôt make logical sense / is outside of the app‚Äôs logical schema?

Cost

  • How much data does the client need to store to fully replicate (full offline data access & editing)? Hopefully O(size of the data) and not O(size of the data + number of changes), with some reasonable constant factor. (this has huge impacts on the ‚Äúinitial app load‚Äù time)

  • How much data does the server need to store? If it needs to store a full change history, does it support periodic garbage collection / compaction?

  • What are the transfer costs of the sync protocol? E.g. are you sending the whole dataset with each change, or just deltas?

  • How‚Äôs the code quality, maintenance level, test coverage, etc.?

Flexibility

  • How does it react to schema changes? If you need to add an attribute to an object, can you?

  • Is the shape of data restricted to anything less than full JSON? e.g. are nested objects, and arrays supported?

  • Can it be integrated into an an existing (server-side or client-side) database (sqlite, postgres, etc.) or do you have to use their specific database?

  • Can it sync with Google Drive, Dropbox, etc. such that each user manages (and pays for) their own backend storage?

  • Does it require all data to live in memory, or can it work with mostly-persisted data? (such that large datasets are usable)

  • Does it support e2e encryption? (zero-knowledge server persistence)

  • Is multi-user collaboration possible?

  • Is collaborative text editing supported? (I‚Äôm fine paying more for it, in terms of server requirements, data overhead, etc.)

  • Does it have the concept of ‚Äúundo‚Äù built-in? At what cost?

  • Does it support a fully p2p network setup (no central authority / server)?

Production-ready

  • How well does it handle offline behavior?

  • Does it correctly handle working on multiple tabs in the same browser session?

  • Does it bake in auth, or can you use an existing authentication setup?
