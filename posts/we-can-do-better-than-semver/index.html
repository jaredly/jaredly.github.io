A vision for validated versioning

Upgrading libraries is both (a) an annoying chore, and (b) has high uncertainty about how ‚Äúsafe‚Äù it is to do. We rely on semantic versioning to signpost when a library undergoes a ‚Äúbreaking change‚Äù, but this is far from reliable.

I think a lot of the issues around versioning & dependency management come down to the fact that versioning is currently ‚Äústringly typed‚Äù: we rely on documentation, readmes, and changelogs (none of which can be statically validated) to indicate versioning guarantees. We can do better.

Side note on surface area

On problem closely tied to versioning is that of ‚ÄúAPI surface area‚Äù. In many languages (Java, Go, NodeJS, OCaml, Elm, Haskell, Koka, Scala), there‚Äôs no way to have a ‚Äúprivate module‚Äù: if you want definitions to be usable in other modules in the package, they also are made available to consumers of the package. In some languages (Python, PHP) you can‚Äôt even have private declarations, so the technical ‚Äúsurface area‚Äù of a library is literally all of your code. (Rust, Swift, and OCaml+Dune stand out as exceptions to this, although none goes as far as I want).

This leaves you reliant on convention and documentation to indicate ‚Äúwhat the publicly-dependable API consists of‚Äù (e.g. ‚Äúdon‚Äôt import the module named Internal, it may change at any time‚Äù).

Wouldn‚Äôt it be nice if our languages actually supported limiting the API surface area of a library?

Automatic determination of breaking changes

Of course, once we‚Äôve locked down the API surface area, there‚Äôs still the problem of determining whether a version includes breaking changes. Why leave that up to changelogs, when we can do better?

Here‚Äôs the vision: breaking changes are statically validatable through a combination of type compatability (what Elm does) and test compatability (I don‚Äôt know of any language that does this).

  1. if the type of a function changes in a non-backwards-compatible way, that‚Äôs a breaking change
  2. if any tests of that function from the previous version fail with the new version‚Äôs implementation, that‚Äôs a breaking change

In this way, the test suite of the public API represents the behavioral guarantees of the library.

This can also allow the package manager to isolate breaking changes to the specific declaration that they impact ‚Äì if your project doesn‚Äôt use that function, then the version isn‚Äôt a breaking change for you.

This also means that version numbers are not an indication of breaking changes; we can instead have them be an indicator of when the version was released, e.g. 20250414.

Now let‚Äôs talk about resolution. The current state of things (with Rust, NPM, Pip) is that a library will specify a range of versions that it is compatible with. In general, this relies on semantic-versioning ranges, which are both too weak (stringly typed) and too restrictive (if the breaking change applies to a method I‚Äôm not using, it shouldn‚Äôt be breaking for me).

What I envision is this: a library A specifies a single version for a dependency B. This is ‚Äúthe version it has been developed against‚Äù. (in this way, it removes some uncertainty around ‚Äúwhat is the /recommended/ version of a thing for a given library‚Äù). But because we have validated versioning, the package manager is free to install a newer version of dependency B, as long as it is determined to be backwards compatible with the requested version, and as long as all of A‚Äôs tests continue to pass.

To add a bit more flexibility, we could have the library also specify a ‚Äúminimum compatible version‚Äù in addition to the ‚Äúrecommended version‚Äù. There are certainly cases where you would want to have a lower version than a library might have been developed against, and it would be good for our dependency metadata to make that easy to do.

How would lockfiles & resolutions work?

(I‚Äôm still at the ‚Äúidea‚Äù stages for this, so I haven‚Äôt fleshed everything out, but here goes)

Libraries would have lockfiles. In the same way that it‚Äôs nice to know the specific version of a dependency that the library was developed against, I would want to download whole exact tree of dependencies that the library was developed against, absent any reason not to (such as a conflicting requirement from another library).

When specifying dependencies (in the equivalent of a package.json or cargo.toml), you would also be able to override the resolution of a transitive dependency, if desired.

How do you deal with ‚Äúmultiple versions of the same package in your tree‚Äù

I‚Äôve run into this issue in JavaScript (multiple versions of React having independent global state that causes errors), Rust (two types from different versions of a library leading to the confusing "found a::b::Foo, expected a::b::Foo", and Java (I don‚Äôt remember why, but my build.gradle has a bunch of exclude group declarations).

I think it ought to be statically analyzeable whether a library‚Äôs use of a dependency is ‚Äúleaky‚Äù. This would either be due to (a) global state being accessed, or (b) some types that are declared in the dependency being ‚Äúvisible‚Äù in the library‚Äôs public API. A ‚Äúleaky‚Äù dependency would therefore need to be locked down so that there‚Äôs only one instance of it between all its consumers, but a ‚Äúnon-leaky‚Äù dependency could be free to have multiple independent versions in use.

Of course, for code size reasons it could be desirable to have more deduplication than the static analysis deems necessary, so we might want to have the ‚Äúisolated dependency‚Äù behavior be opt-in instead of opt-out.

Migrations, can we have them

We can take our ‚Äúbreaking changes‚Äù story one step further by introducing migration macros. There is a class of breaking changes that can be alleviated by making a relatively simple source code transformation of each call site. Examples include: removal of an unused function parameter, changing the type of an argument from int to Option<int>, swapping the order of two arguments of the same type, etc.

When publishing a new version of your library, you would be able to specify migration macros for public API declarations that have had such signature changes, turning what would otherwise be a breaking change into a non-breaking one.

What if nothing ever breaks?

A further extension of these ideas that I‚Äôm considering is to just never allow breaking changes. This is in line with Golang‚Äôs ethos, where a ‚Äúnew major version‚Äù of a package is treated as a ‚Äúnew package‚Äù: instead of ‚Äúauth‚Äù, we‚Äôre using ‚Äúauth/v2‚Äù. There‚Äôs a definite appeal to this, in that it encourages a high commitment to compatability, while also allowing for a clean break when such is necessary. (in the absence of migrations, such a policy would be needlessly restrictive, so it‚Äôs a good thing we‚Äôve accounted for them)

One issue this would hopefully eliminate is the need to handle ‚Äúun-breaking‚Äù versions; e.g. if version 20250411 had a breaking change, and then 20250412 ‚Äúreverted‚Äù that breaking change, it would be great if a package manager were able to ‚Äúknow‚Äù that it can just skip over 20250411 and continue with compatible upgrades. This could add quite a bit of complexity however, and so it‚Äôs appealing to just ‚Äúeliminate casual breaking changes‚Äù out of hand.

FAQ

Why are you playing around with dependency management of a single language? You should be working on whole-system (cross-language) dependency management a la Nix.

I dunno, I want to have nice things? And a whole-system package manager can only be as smart as the dumbest language. I want smarter languages.

Packages should be hand-curated, not managed by soulless machines

I believe that the main critiques of package management are attributable to (1) having a language that is oblivious to the notion of a package, and (2) critiques of human-curated semver, which is precisely what I‚Äôm hoping to supplant. There‚Äôs also (3), ‚Äútrust no one, the world will betray you‚Äù, and that‚Äôs a social problem that it seems the Elm community is doing a good job of combatting.

Doesn‚Äôt this require the package-manager / compiler to be able to run the tests for a library? Should we really be baking testing into the language itself?

Yes, yes it does. If you‚Äôre building a language that doesn‚Äôt have unit testing built into it, what are you playing at.
